# v2.1 re-extract — H2 source filter + H6 determinism

**Dataset version:** v2.1 &nbsp;·&nbsp; **Cutoff date:** 2026-05-28 &nbsp;·&nbsp;
**Sources:** Harvard + Columbia + Princeton (LoC dropped, 2014 cap)

This re-extract regenerated `data/v2/**` from the bulk MARC with two fixes that
require re-reading the corpus:

- **H2 — MARC 6xx source filter** (`scaleup/marc_index.py:_is_lcsh_heading`).
  Keep only headings actually cataloged as LCSH/LCGFT; drop FAST, MeSH, and
  other thesauri. Rule:
  - `600/610/611/630/650/651`: keep iff `ind2==0` (LCSH) **or** (`ind2==7` and
    `$2==lcsh`).
  - `655`: keep iff `$2==lcgft` (LCGFT).
- **H6 — deterministic stratification** (`scaleup/match.py`): `lang`/`lc_class`
  chosen by fixed source precedence, so the sampler is reproducible.

The re-extract ran locally from the cached raw MARC (`data/raw/v2/raw_cache` +
`incoming/harvard`), no re-download. Procedure: `docs/vps-reharvest-runbook.md`.

### Why −25.9% removed but the H2 audit predicted ~8.4%

The pre-extract audit (`marc-leakage-audit.md`) measured **string validity**
(GT roots absent from the LCSH+LCGFT vocab → ~8.4%). H2 enforces **provenance**
(cataloged as `ind2==0` LCSH). FAST is *derived from* LCSH, so FAST headings
have valid-LCSH strings (vocab-reachable → not flagged by the audit) yet are
assigned `$2=fast` → H2 drops them. The 25.9%−8.4% gap is precisely those
FAST/bisacsh/etc. look-alikes. This is the agreed provenance-based design (GT =
what was cataloged as LCSH, not what *could* be LCSH), not a defect. Keeping
FAST-derived valid-LCSH strings is the lever if this is ever revisited.

### Non-Latin fraction rose (0.82% → 1.00%) despite cleaning

Absolute non-Latin GT fell (856 → 770). FAST/MeSH are overwhelmingly
English/Latin, so removing them shrinks the Latin denominator faster than the
(CJK/Cyrillic) numerator, raising the *relative* share of the non-Latin LCSH
that legitimately remain.

## Baseline re-score

Only the **frequency-floor** baseline was re-scored on this machine (it is
input/GT-driven, no vocab/index/API needed). The cleaner GT lowers the floor —
the v2.0 numbers were inflated by frequent non-LCSH matches.

| set | metric | v2.0 | v2.1 |
|---|---|---:|---:|
| dev (19,053) | recall@200 exact | 0.128 | 0.072 |
| dev | MRR exact | 0.186 | 0.041 |
| test (3,363, hashed) | recall@200 exact | 0.128 | 0.067 |
| test | MRR exact | 0.194 | 0.041 |

The dev-2K **subset** frequency floor was also corrected (a stale subset
`.score.json` had carried a v2.0-era 0.092); the true v2.1 subset floor is
recall@200 exact **0.075** / MRR **0.044**, consistent with full dev (0.072) as
a stratified sample should be. All three frequency prediction files were
regenerated to 100% v2.1 record-ID overlap before re-scoring.
